# 文档规范

`Sunny Docs` 不只是展示 Demo，而是组件库对外的产品说明书。文档规范的目标是让每个组件页结构一致、信息完整、维护成本低。

## 文档目录约定

```text
apps/docs
  components/
    demos/         # 组件 Demo
    *.md           # 组件说明页
  guide/           # 规范与接入说明
  .vitepress/      # 站点配置与主题
```

## 组件文档页结构

每个组件页默认遵循下面的顺序：

1. Demo
2. Props
3. Events
4. Slots
5. 用法示例

如果某个组件没有 `Events` 或 `Slots`，可以省略对应章节，但顺序尽量保持一致。

## Demo 编写原则

- Demo 文件放在 `components/demos/`
- 一个 Demo 只讲一个核心场景
- 标题说明“这是什么”，描述说明“适合在哪用”
- 示例代码要与推荐导入方式一致

## 文案规范

- 优先解释语义和适用场景，不只描述视觉
- 避免“就是一个按钮”这类无信息量描述
- 面向团队协作写文档，默认读者既包括组件维护者，也包括业务使用者

## 站点配置规范

- 导航与侧边栏优先来自组件注册表
- `guide/` 放团队规范和接入说明
- `components/` 放稳定对外的公共组件

## 文档更新 checklist

1. 组件 API 变更后同步更新 Props / Events / Slots
2. 如果推荐导入方式变化，更新示例代码
3. 如果组件分类变化，更新注册表
4. 重新运行 `pnpm build:docs`
5. 检查 `Sunny Studio` 的真实页面是否仍然与文档一致
